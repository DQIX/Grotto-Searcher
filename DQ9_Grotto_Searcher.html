Rank、Seed

藏寶圖表面上是由 地圖名稱 組成，實際上是由 Rank 和 Seed 兩個隱藏參數構成。
本工具的核心運作原理，就是透過您輸入的各種表面特徵，查出 Rank 與 Seed 這兩個代表每張地圖的「身分證號」。

Seed

  • 合法取值範圍為 0000 ~ 7FFF。
  • 雖然系統中存在 8000 ~ FFFF 的 Seed，但在正常遊戲中沒有合法的生成手段。
  • 即使在合法範圍內，仍有 302 個 特定 Seed 無法被正常生成。
  • 所有不能合法生成的地圖，本工具皆會將其判定並顯示為「無符合條件的位置」。

Base Quality (BQ)、Final Quality (FQ)

Base Quality (BQ)
BQ = 主角全職業最高Lv + 主角全職業最高轉生次數×5 + 正在攻略的地圖Lv
Final Quality (FQ)
FQ = BQ±10%
FQ 最小為 2，最大為 248。地圖的 FQ 數值落在哪個區間，該地圖的 Rank 就是多少。
📖 地圖生成的詳細三步曲

遊戲生成地圖時，會使用一個隨時間跳動的 Timer (0 ~ 65535)，並將其送入 AT 函數 (LCG) 連續推進 3 步：
LCGn+1 = [1103515245×LCGn+12345] % 232 再將其高 16 bit 轉換為 15 bit

  1. 決定 FQ 浮動值
    跑第一遍 AT 函數，作為「隨機數」代入以下公式算出「±10%」的具體值，進而算出 FQ 與 Rank：
    FQ = BQ + int(隨機數 % [int(BQ÷10)×2+1] - BQ÷10)
    ※ 程式碼中的兩次 int() 捨去小數點動作皆不可省略。
  2. 決定地圖 Seed
    跑第二遍 AT 函數，直接用來作為這張地圖的 Seed。
  3. 決定地圖的入口位置
    跑第三遍 AT 函數，根據算出的 FQ 查表決定這張地圖「能出現的有效位置總數」(例如 FQ 2~50 時有效位置為 47 個)，再將這次 AT 對「有效位置總數」取餘數 (modulo, 也就是上面的 %)，由此決定最終在大地圖上生成地圖入口的具體位置代碼。

地形

地形共分為 洞窟、遺跡、冰、水、火山 五種。生成機率分別為：洞窟佔 30%，遺跡佔 40%，其餘各佔 10%。部分 QUEST 需要獲取「指定地形」與「指定 FloorMR」的地圖才能達成。

  • 遺跡：十字路口出現怪物時 極易被堵死。
  • 洞窟 & 水：廣闊空間的邊緣有突出結構，會讓玩家多走一點腳程。
  • 洞窟 & 冰：廣闊空間有像小石子一樣的障礙物，移動時需注意避開。
  • 洞窟 (FloorMR 10-11)：會出現能把所有路口、通路 全部堵死 的「だいおうクジラ」，嚴重影響玩家從入口快速走到 BOSS 面前的體驗，其他地形問題沒那麼嚴重。
  • BOSS 樓層所需時間：從樓層入口到 BOSS 跟前的所需時間也有地形差別：遺跡 6.6秒，火山 7.4秒，其餘皆為 7.2秒。這微小的差距足以改變部分低Lv發掘用地圖的排名。(來源: 勇気ある旅人さんwiki)

📖 實際的生成機制

乳首、イケない通路、チラリズム

　地圖生成時，系統先用二元空間分割法把樓層切成多個區域，在每個區域裡挖出房間，再沿著分割線畫走廊連通所有房間。接著進入「侵蝕」階段：根據每個格子周圍牆壁的數量決定它是保留為地板還是填回牆壁。侵蝕不知道哪些走廊是唯一的通道，所以可能把某條 1 格寬的走廊填成牆壁，導致部分房間被完全隔離。之後放寶箱時，系統從房間的邊界範圍內隨機抽座標，只跳過已有樓梯或寶箱的格子，不檢查該格子是不是牆壁、也不檢查玩家能不能走到。如果寶箱落在四面皆牆的位置，就嵌在牆壁正中央，玩家完全看不見——這就是「乳首」。如果走廊被侵蝕切斷但格子本身保留了下來，就成為一段被牆壁封死、無法進入的白色通道——這就是「イケない通路」。「チラリズム」則是指一段 1x2 或以上的「イケない通路」裡面有寶箱的情形。

幽靈樓梯、危険？ハマるフロア

　放置樓梯時，系統先隨機抽一個 已經通過所有檢查的 上樓梯候選位置，寫入地圖後再抽下樓梯候選。如果下樓梯被拒（例如卡在角落），系統回到外層重新抽上樓梯——但上一次寫入的上樓梯不會被清除，殘留在地圖中成為「幽靈樓梯」：外觀是普通地板，底層數據卻是樓梯。大部分情況下幽靈樓梯只是無害的殘留。但當地圖結構極端受限、100 次候選全部無法通過檢查時，系統進入溢出模式：跳過所有檢查直接放上樓梯，然後將計數器歸零重來。如果下樓梯再次失敗，需要再累積 100 次拒絕才能再次溢出。每次溢出都可能留下新的幽靈樓梯，而這些幽靈樓梯還會污染鄰居格子的判定，導致後續的下樓梯候選也連帶失敗。最終定案的樓梯位置可能出現上樓梯緊鄰寶箱的佈局，形成一道玩家無法穿越的障壁，把地圖分成兩個不連通的區域——走下去就被困住，走上去就回不來。這就是「危険？ハマるフロア」。
　一個樓層 100 次候選全部無法通過檢查，需滿足以下條件：所有廣域空間都是 2x2，除此之外全是走廊。
　並不是每一個這種類型的樓層都會產生樓梯與寶箱緊鄰的佈局。5BC3 B4F / 5BC4 B3F 會產生這種佈局實在是非常特殊。

被封在牆壁中的樓梯

　根據上述機制，如果一個樓層 100 次上樓梯候選全部無法通過檢查，進入溢出模式後跳過所有檢查，就有可能把上樓梯也放進牆壁。但由於 100 次候選全部無法通過檢查的樓層實在太少，總共就 4 種 16 個搜尋結果，實際上這種情形並沒有發生。

📖 實際機制 與 dq9tmap101.exe 誤判的歷史

記憶體溢出 導致超大面積樓層的出現怪物減少

　DQ9 如果一層樓面積特別大、白方格特別多，會使樓層面積的數據變得龐大，過度佔據記憶體，而把每層樓末端的數據從記憶體中「擠出去」。由於末端數據剛好用來存放該樓層的「怪物生成列表 (Enemy List)」，一旦被擠掉，能出現的怪物種類就會變少。少了一種或幾種叫做 敵減，少到僅剩一種叫做 ONLY，全部被擠掉叫做 敵無。

衍生 BUG

  • 寶箱怪消失 BUG：寶箱怪佔了數據位置但不在地圖上出現。若有怪物 ID 排在寶箱怪之前 (例如はぐれメタル)，則會導致在這些 ONLY 地圖中出現寶箱怪「一生成就立即消失」的奇異現象。
  • 吹口哨失敗 BUG：使用戰士技能「くちぶえ」時，如果系統抽到的怪物剛好是「被擠掉的那幾種」，就會發生呼叫不出怪物的判定失敗。
  • Flag0 方格 與 掉寶操縱：某些大面積樓層因白方格過多，導致白方格專用的記憶體空間也發生溢出。這會使該樓層右下角出現「上下左右鄰接四邊不為零，卻被強制當作零處理」的白方格 (即 Flag0)。在此方格內不會出現怪物，怪物也不會走進來（除非追蹤玩家）。但若在此處使用「くちぶえ」，不僅能正常呼叫怪物，且 AT 亂數在進入戰鬥後會保持停止狀態。 這個特質被極致運用於「地圖法秘技」來精準操縱怪物掉寶。

「ONLY」誤判為「敵無」「無無無」的歷史

　此機制最初由 dq9tmap101.exe 的作者 (43) 發現，並命名為 ElistOfs (Enemy List Offsets)。但當時作者並未發現 Flag0 的存在，導致所有帶 Flag0 的地圖算出來的 ElistOfs 比真實值大很多，產生了大量的誤判（例如「敵減」被誤判為「ONLY」，「ONLY」被誤判為「敵無」）。
　除此之外，作者還犯了一個人為錯誤——當所有白方格鄰接四邊之和的總數÷2 ≧ 256 時，程式莫名其妙地把它減去 256，造成了「ElistOfs=1Exx~1Fxx 但卻是特殊樓層」的嚴重誤判 (也就是「敵無/敵減/ONLY 誤判為 無無無」)。
　由於作者後來遭到 5ch 網友排擠，dq9tmap101.exe 停止更新，這個誤判就這樣存在了 16 年。在此期間，Terence 於 2010 年 10 月發表的 Grotto Mechanics Guide 初步提到了白方格過多導致的怪物不生成現象，但沒有進一步研究。而日本玩家群則靠著一張張地圖打出來比對，建立了不少資料庫 (例如 オンリー敵減(2種) wiki，ぶぶづけ さん為 dq9tmap101.exe 做的 mapdata patch (網站已於 2026.3.31 關閉) 等)，但沒人從機制上探索。
　2025年12月，本工具的開發者委託 TKG 徹底分析了 dq9tmap101.exe，終於把 ElistOfs 的真相找出來了：

ElistOfs 計算公式

  1. 令 Boundary = 4128 - 16W - 8X
    W：該樓層白方格總數
    X：從左到右、從上到下 逐一檢查該樓層每個白方格 得到的 L+U 的總和 (即 X = L + U)
    L：若該白方格的左邊一格也是白方格，L = 1，否則 L = 0
    U：若該白方格的上邊一格也是白方格，U = 1，否則 U = 0
  2. Boundary≥0 時，A = 4896 + 16W + 8X
    Boundary＜0 時，A = 4896 + 16W + 8B
    B = (9016 - 4896 - 16W) ÷ 8 = (4120 - 16W) ÷ 8 = 515 − 2W
  3. C = A + 4 + 8B + 4D
    D：令 D 的初始值為 0，B_pool 的初始值為 B
    再次從左到右、從上到下 逐一檢查每個白方格
    若進入該白方格時 B_pool ≤ 0，則 D 加 1，不做任何扣減，直接跳到下一個白方格
    若 B_pool > 0：
    先看 L：若 L = 1，B_pool 減 1（此步不檢查 B_pool 是否已歸零）。
    再看 U：若 U = 1 且此時 B_pool > 0，B_pool 再減 1。
    檢查完所有白方格後，D 即為「B_pool 耗盡後仍有連通邊的白方格數量」
  4. E = C + 20F
    F：該樓層出現的怪物種類數 (冰 1/10~12、遺跡 3：F = 7，其他：F = 8)
  5. ElistOfs = E + 8F + G + 海外版 Padding
    G：該樓層出現的怪物 的固有值之和
    海外版 Padding：海外版 DQ9 為 128，日本版 DQ9 為 0

Rank & Seed

A treasure map is nominally defined by its grotto name, but internally it is constructed from two hidden parameters: Rank and Seed.
The core principle of this tool is to determine the Rank and Seed — the "ID number" of each map — based on the various nominal features you input.

Seed

  • The valid range is 0000 ~ 7FFF.
  • Although Seeds in the range 8000 ~ FFFF exist in the system, there is no legitimate way to generate them during normal gameplay.
  • Even within the valid range, there are still 302 specific Seeds cannot be generated.
  • All maps that cannot be legitimately generated will be identified and displayed by this tool as "No Location."

📖 The 3-step map generation process in detail

When the game generates a map, it uses a Timer (0 ~ 65535) and puts it into the AT function (LCG) for 3 consecutive steps:
LCGn+1 = [1103515245×LCGn+12345] % 232
Then convert its upper 16 bits to 15 bits: (LCGn+1 >>> 16) & 0x7FFF

  1. Determine the actual value of ±10%
    Run the AT function for the first time, use the result as the RNG, and put it into the following formula to calculate the actual value of ±10%, thereby obtaining FQ and Rank:
    FQ = BQ + int(RNG % [int(BQ÷10)×2+1] - BQ÷10)
    ※ Both int() truncations in the code are essential and must not be omitted.
  2. Determine the map Seed
    Run the AT function for the second time, and use the result directly as this map's Seed.
  3. Determine the map entrance location
    Run the AT function for the third time. Based on the calculated FQ, get the total amount of its Location Range's locations (e.g., when FQ is 2~50, its Location Range is 01~2F which contains 47 locations). Then use this AT result to compute the modulo (%) against the "location's total amount," and the result determines the specific location code where the map entrance is placed on the overworld.

Environment

There are 5 terrain types: Caves, Ruins, Ice, Water, and Fire. Caves has 30%, Ruins has 40%, and the rest have 10% each. Certain quests require maps with a specific environment and a specific FloorMR to complete.

  • Ruins: When a monster spawns at the crossroad, it is easy to get blocked.
  • Caves & Water: The edges of open areas have protruding structures, forcing the player to walk a bit.
  • Caves & Ice: Open areas have small pebble-like obstacles that require careful avoidance when moving.
  • Caves (FloorMR 10-11): The monster Stale Whale can completely block all junctions and corridors, severely hindering the player's ability to quickly reach the BOSS from the entrance. Other environments have less severe issues.
  • BOSS floor travel time: The time from the floor entrance to the BOSS also varies by environment: Ruins 6.6s, Fire 7.4s, all others 7.2s. This small difference is enough to affect the ranking of some low-Lv excavation maps. (Source: 勇気ある旅人さんwiki)

📖 Mechanism

Nipple, Chamber

　During map generation, the system first uses Binary Space Partitioning (BSP) to divide the floor into multiple regions, carves out a room in each region, and then draws corridors along the partition lines to connect all rooms. Next comes the "erosion" phase: based on the number of surrounding walls for each tile, the system decides whether to keep it as floor or fill it back in as wall. Erosion is unaware of which corridors are the only passageway, so it may fill in a 1-tile-wide corridor, completely isolating some rooms. When placing chests afterward, the system randomly picks coordinates within a room's boundary, skipping only tiles that already contain stairs or chests — it does not check whether the tile is a wall, nor whether the player can reach it. If a chest lands on a tile surrounded by walls on all four sides, it becomes embedded in the center of a wall, completely invisible to the player — this is Nipple (乳首). If a corridor is severed by erosion but the tiles themselves are preserved, it becomes a white passage sealed off by walls that cannot be entered — this is Chamber (イケない通路). The Japanese term チラリズム (Nipple+Chamber) refers to the case where a 1x2 or larger Chamber contains a chest inside it.

Ghost Stairs, Softlock

　When placing stairs, the system first randomly selects an up-staircase candidate position that has passed all checks, writes it to the map, and then selects a down-staircase candidate. If the down-staircase is rejected (e.g. stuck in a corner), the system goes back and re-selects the up-staircase — but the previously written up-staircase is unwiped, remaining in the map as a "Ghost Staircase": it looks like an ordinary floor tile, but the underlying data says it is stairs. In most cases, ghost staircases are just harmless remnants. However, when the map structure is extremely constrained and all 100 candidates fail the checks, the system enters overflow mode: it skips all checks and places the up-staircase directly, then resets the counter to zero and starts over. If the down-staircase fails again, another 100 rejections must accumulate before overflow triggers again. Each overflow may leave behind a new ghost staircase, and these ghost staircases can contaminate the adjacency checks of neighboring tiles, causing subsequent down-staircase candidates to fail as well. The final staircase placement may result in an up-staircase placed right next to a chest, forming an impassable barrier that splits the map into two disconnected areas — going down traps you, going up leaves no way back. This is Softlock (危険？ハマるフロア).
　For a floor to have all 100 candidates fail, the following condition must be met: all open areas are 2x2, and everything else is corridors.
　Not every floor of this type produces a stairs-adjacent-to-chest layout. The fact that 5BC3 B4F / 5BC4 B3F produce this layout is truly exceptional.

Wall-Embedded Stairs

　Based on the mechanism described above, if all 100 up-staircase candidates on a floor fail the checks and the system enters overflow mode (skipping all checks), it would theoretically be possible for an up-staircase to be placed inside a wall. However, since floors where all 100 candidates fail are exceedingly rare — only 4 layout types yielding 16 search results in total — this situation has never actually occurred.

📖 Mechanism & History of dq9tmap101.exe Misjudgments

Wandering Monster Floor is caused by Memory Overflow

　In DQ9, if a floor has a very large area with many white tiles, the floor layout data becomes so large that it overflows its memory allocation, pushing out the data stored at the end of each floor's memory block. Since this tail-end data happens to store the floor's "Enemy List," the types of monsters that can appear are reduced. Losing a few types is called 2-enemy / 3-enemy / 4-enemy, being left with only one type is called ONLY, and losing all types is called No-enemy.

Derived Bugs

  • Vanishing Trap Monster Bug: Trap monsters (Mimic, etc.) occupy data slots but do not actually appear on the map. If a monster ID is listed before the trap monster (e.g. Liquid Metal Slime), it causes the trap monster to vanish immediately when it spawns on these special floors.
  • Whistle Failure Bug: When using the Warrior skill "Whistle" (くちぶえ), if the system selects a monster type that has been "pushed out," the Whistle fails to summon it.
  • Flag0 Tiles & Drop Manipulation: On some large floors, the sheer number of white tiles also overflows the memory space dedicated to white tile adjacency data. This causes certain white tiles in the bottom-right corner to have their adjacency flags forced to zero despite actually having nonzero neighbors (these are Flag0 tiles). No monsters spawn inside these tiles, and monsters will not walk into them (unless chasing the player). However, if you use "Whistle" while standing on a Flag0 tile, monsters not get pushed out are summoned normally, and AT freezes after entering battle. This property is exploited to its fullest in the "Map Method" technique for precise drop manipulation.

History: "ONLY" misjudged as "No-enemy" / "Triple None"

　This mechanism was originally discovered by the author of dq9tmap101.exe (known as "43"), who named it ElistOfs (Enemy List Offsets). However, the author was unaware of the existence of Flag0, which caused the computed ElistOfs for all maps with Flag0 tiles to be much larger than the true value, leading to widespread misjudgments (e.g. "2-enemy" misidentified as "ONLY," "ONLY" misidentified as "No-enemy").
　Additionally, the author made a coding error: when the sum of all white tile adjacency values ÷ 2 was ≥ 256, the program inexplicably subtracted 256, causing severe misjudgments where "ElistOfs = 1Exx–1Fxx maps were actually special floors" (i.e. "No-enemy / Reduced / ONLY" misjudged as "Triple None").
　After the author was ostracized by 5ch users, dq9tmap101.exe ceased updates, and these misjudgments persisted for 16 years. During this period, Terence published the Grotto Mechanics Guide in October 2010, briefly mentioning the phenomenon of monsters not spawning due to excessive white tiles, but did not investigate further. Japanese players instead built databases through brute-force play-testing (e.g. the オンリー敵減(2種)wiki, ぶぶづけ's mapdata patch for dq9tmap101.exe (site closed on 2026.3.31), etc.), but no one explored the underlying mechanism.
　In December 2025, this tool's developer commissioned TKG to conduct a thorough analysis of dq9tmap101.exe, finally uncovering the truth behind ElistOfs:

ElistOfs Calculation Formula

  1. Let Boundary = 4128 − 16W − 8X
    W: Total number of white tiles on the floor
    X: Scanning every white tile left-to-right, top-to-bottom, the sum of L + U for all tiles (i.e. X = ΣL + ΣU)
    L: If the tile immediately to the left is also a white tile, L = 1; otherwise L = 0
    U: If the tile immediately above is also a white tile, U = 1; otherwise U = 0
  2. If Boundary ≥ 0: A = 4896 + 16W + 8X
    If Boundary < 0: A = 4896 + 16W + 8B
    B = (9016 − 4896 − 16W) ÷ 8 = (4120 − 16W) ÷ 8 = 515 − 2W
  3. C = A + 4 + 8B + 4D
    D: Initialize D = 0, B_pool = B
    Scan every white tile left-to-right, top-to-bottom again
    If B_pool ≤ 0 upon entering a tile: increase D by 1, skip to the next tile without any deduction
    If B_pool > 0:
    Check L: if L = 1, decrease B_pool by 1 (without checking whether B_pool has reached zero at this step)
    Check U: if U = 1 and B_pool > 0 at this point, decrease B_pool by 1
    After all white tiles are checked, D = "the number of white tiles that still had neighboring connections after B_pool ran out"
  4. E = C + 20F
    F: Number of monster types on the floor (Ice MR 1/10–12, Ruins MR 3: F = 7; otherwise F = 8)
  5. ElistOfs = E + 8F + G + International DQ9 Padding
    G: Sum of the intrinsic values of all monsters on the floor
    International DQ9 Padding: 128 for non-JP DQ9, 0 for JP DQ9

📖 実際の生成メカニズム

乳首、イケない通路、チラリズム

　地図を生成する時、システムはまず「バイナリ空間分割法 (BSP)」でフロアを複数の領域に分割し、各領域内に部屋を掘り出した後、分割線に沿って通路を引いてすべての部屋を接続する。次に「侵食」フェーズに入る：各マスの周囲にある壁の数に基づき、そのマスを床として残すか壁に埋め戻すかを決定する。侵食はどの通路が唯一の経路であるかを認識しないため、通路の入口を壁で埋めてしまい、その中の部屋を完全に孤立させることがある。その後、宝箱を配置する際、システムは部屋の境界範囲内からランダムに座標を選び、既に階段や宝箱があるマスのみスキップする——つまりそのマスが壁かどうか、プレイヤーが到達可能かどうかを一切チェックしない。宝箱が四周に壁に囲まれたマスに配置されると、壁の中に埋め込まれ、ゲームの中から完全に見えない、あるいは空中に浮いているように見える——これが「乳首」。入口が侵食により切断され、壁に封じ込められた進入不可能な通路となる場合——これが「イケない通路」。「チラリズム」とは、1x2 以上の「イケない通路」の中に宝箱がある状態を指す。

ゴースト階段、危険？ハマるフロア

　階段を配置する際、システムはまず「全てのチェックをパスした」上り階段の候補マスをランダムに選び、マップに上り階段を書き込んだ後、下り階段の候補マスを選ぶ。下り階段のチェックが失敗になった場合（例：そのマスは隅である場合）、システムは上り階段を改めて選び直す——しかし前回書き込んだ上り階段は消されず、マップ内に「ゴースト階段」として残留する：見た目は普通の床マスだが、内部データは階段である。つまり「ゴースト階段＝上り階段にマッチングされた下り階段がチェック失敗の時に残された上り階段の残骸」である。ほとんどの場合、ゴースト階段は無害な残留物に過ぎない。しかし、マップの構造が極端に制限され、100 回連続で上り階段のチェックをパスできない場合、システムはオーバーフローモードに入る：全てのチェックをスキップして直接に上り階段を配置し、上り階段のチェック回数のカウンターをゼロにリセットし、下り階段のチェックモードに入る。下り階段もチェックが失敗した場合、マップ構造の制限により、次の上り階段配置フェースまで、また99回連続 (合計100回) で上り階段のチェックを失敗し、オーバーフローを再度誘発することになる。つまり「オーバーフローが発生する場合、上り階段を配置するチャンスは100回ごとに1回、ゴースト階段として残す場合も100回ごとに1回」。また、ゴースト階段は隣接マスの判定を影響し、その後の下り階段チェックも連鎖的に失敗させる。最終的な階段配置では、上り階段が宝箱の真隣りに置かれるレイアウトが生じることがあり、プレイヤーが通過できない障壁を形成して地図を 2 つの非連結エリアに分断する——下に行けば閉じ込められ、上に行けば戻れない。これが「危険？ハマるフロア」である。
　100 回連続でチェックをパスできないフロアの条件：全ての広い空間が 2x2 で、それ以外は全て通路であること。
　ただし、この条件を満たすフロアの全てが「階段と宝箱が隣接する」レイアウトを生成するわけではない。5BC3 B4F / 5BC4 B3F でこのレイアウトが発生するのは極めて特殊なケースである。

壁に埋まった階段

　上記のメカニズムに基づき、あるフロアで 100 回連続で上り階段のチェックをパスできず、オーバーフローモードに入った場合、上り階段が壁の中に配置される可能性は理論上ある。しかし、100 回連続でチェックをパスできないフロアは極めて少なく、全部で 4 種 16 件の検索結果しかないため、実際にはこの現象は発生していない。

特殊フロアの産生原因：メモリオーバーフロー

　DQ9 では、あるフロアの面積が非常に大きく白マスが多い場合、フロアレイアウトのデータが肥大化してメモリ領域を圧迫し、各フロアの末端に格納されたデータを「押し出して」しまう。この末端データはそのフロアの「モンスター生成リスト (Enemy List)」の格納場所であるため、押し出されると出現可能なモンスターの種類が減少する。数種類減ったものを 敵減、1 種類だけ残ったものを オンリー、すべて押し出されたものを 敵無 と呼ぶ。

派生バグ

  • トラップモンスター消失バグ：トラップモンスター（ミミック等）はデータスロットを占有するが、マップ上には出現しない。もしトラップモンスターより前にモンスター ID が配置されている場合（例：はぐれメタル）、トラップモンスターが「出た瞬間に消滅する」という奇妙な現象が発生する。
  • 「くちぶえ」失敗バグ：戦士のスキル「くちぶえ」を使用した際、システムが「押し出された」モンスターを選択すると、モンスターを呼び出せない判定失敗が発生する。
  • Flag0 マスと地図法：一部の大面積フロアでは、白マスが多すぎるため、白マス専用のメモリ領域もオーバーフローする。その結果、右下の白マスが「上下左右の隣接辺がゼロでないにもかかわらず、強制的にゼロとして処理される」状態になる（Flag0 マス）。このマス内にはモンスターが出現せず、モンスターも入ってこない（プレイヤーを追跡中の場合を除く）。しかし、Flag0 マス上で「くちぶえ」を使うと、モンスターは正常に呼び出され、戦闘に入った後も AT 乱数が停止した状態を維持する。 この特性は「地図法」でモンスターのアイテムをドロップさせることに最大限活用されている。

「オンリー」を「敵無」「無無無」と誤判定した歴史

　このメカニズムは dq9tmap101.exe の作者（43）氏によって最初に発見され、ElistOfs (Enemy List Offsets) と命名された。しかし、当時の作者は Flag0 の存在に気づいておらず、Flag0 マスを持つすべての地図で算出された ElistOfs が実際の値より大幅に大きくなり、大量の誤判定を引き起こした（例：「敵減」が「オンリー」と誤判定し、「オンリー」が「敵無」と誤判定した）。
　さらに、作者はコーディング上のミスも犯していた——全ての白マスの隣接辺の合計値 ÷ 2 が 256 以上の場合、プログラムがなぜか 256 を減算しており、「ElistOfs = 1Exx〜1Fxx だが実際は特殊フロア」という深刻な誤判定が発生した（すなわち「敵無/敵減/オンリー を 無無無 として誤判定した」）。
　作者がその後 5ch ユーザーからハブられたことで dq9tmap101.exe の更新は停止し、この誤判定は 16 年間そのまま放置された。この間、Terence 氏が 2010 年 10 月に発表した Grotto Mechanics Guide で白マス過多によるモンスターの出ない現象に簡単に触れたが、それ以上の調査は行われなかった。日本のプレイヤーコミュニティは地図を一枚ずつ発掘して照合し、データベースを構築した（例：オンリー敵減(2種) wiki、ぶぶづけ 氏による dq9tmap101.exe 用 mapdata パッチ（サイトは 2026.3.31 に閉鎖）など）が、メカニズムの解明には至らなかった。
　2025 年 12 月、本ツールの開発者が TKG 氏に依頼して dq9tmap101.exe を徹底的に分析した結果、ついに ElistOfs の真相が解明された：

ElistOfs の計算式

  1. Boundary = 4128 − 16W − 8X とする
    W：そのフロアの白マス総数
    X：左から右、上から下の順に各白マスを走査し、L + U の合計を求める（X = ΣL + ΣU）
    L：左隣のマスも白マスなら L = 1、そうでなければ L = 0
    U：上隣のマスも白マスなら U = 1、そうでなければ U = 0
  2. Boundary ≥ 0 の場合：A = 4896 + 16W + 8X
    Boundary < 0 の場合：A = 4896 + 16W + 8B
    B = (9016 − 4896 − 16W) ÷ 8 = (4120 − 16W) ÷ 8 = 515 − 2W
  3. C = A + 4 + 8B + 4D
    D：D の初期値を 0、B_pool の初期値を B とする
    再び左から右、上から下の順に各白マスを走査する
    あるマスに入った時点で B_pool ≤ 0 なら、D を 1 加算し、何も減算せず次のマスへ進む
    B_pool > 0 の場合：
    まず L を確認：L = 1 なら B_pool を 1 減算（この時点で B_pool がゼロになったかは確認しない）
    次に U を確認：U = 1 かつこの時点で B_pool > 0 なら、B_pool をさらに 1 減算
    すべての白マスを確認した後、D =「B_pool が枯渇した後もなお隣接辺を持つ白マスの数」
  4. E = C + 20F
    F：そのフロアに出現するモンスター種類数（氷 MR 1/10〜12、遺跡 MR 3：F = 7、その他：F = 8）
  5. ElistOfs = E + 8F + G + 海外版 Padding
    G：そのフロアに出現するモンスターの固有値の合計
    海外版 Padding：海外版 DQ9 は 128、日本版 DQ9 は 0
